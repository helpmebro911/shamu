# Phase 8.A — Linear daemon soak test

The Phase 8.A autonomous-loop story requires the daemon to run unattended
for 24 hours without leaks, wedge, or unbounded memory growth. Two soak
harnesses cover that goal:

1. **Fast soak** — time-compressed in-process soak with a fake
   `LinearClient` + fake webhook iterator. Runs in ≤60s on CI.
2. **Real soak** — 24h run against a staging Linear project with a real
   cloudflared tunnel. Operator-driven; not part of CI.

## 1. Fast soak (local / CI)

Entry point: `apps/cli/scripts/soak-daemon.ts`.

```
bun apps/cli/scripts/soak-daemon.ts \
  --duration=60s \
  --rate=1 \
  --global-cap=3 \
  --queue-capacity=8 \
  --telemetry=/tmp/soak.ndjson
```

Flags:

| Flag                   | Meaning                                         | Default |
|------------------------|-------------------------------------------------|---------|
| `--duration=<N[ms|s|m|h]>` | Total run length.                           | `60s`   |
| `--rate=<N>`           | Synthetic `shamu:ready` events per second.      | `1`     |
| `--global-cap=<N>`     | Rate-limiter `globalCap`.                       | `3`     |
| `--queue-capacity=<N>` | Rate-limiter bounded-queue capacity.            | `8`     |
| `--telemetry=<path>`   | NDJSON output path (defaults to stdout).        | stdout  |

Exit codes: `0` clean, `1` invariant violation, `2` unexpected error.

### Invariants checked

- `inFlight <= globalCap` at every 5s tick.
- `queueLength <= queueCapacity` at every tick.
- Heap usage stays below `4×` baseline (or within 64 MiB of baseline if
  the baseline is tiny).
- Every pickup completes (no stuck in-progress) by the drain at EOS.

### NDJSON events

`start`, `tick` (every 5s), `drain_start`, `done`. `done` carries the
final invariants snapshot + any `violation` record.

Parse with `jq`:

```
jq -c 'select(.kind=="tick") | {elapsed:.elapsedMs,inFlight,peakInFlight,heap:.heapBytes}' /tmp/soak.ndjson
```

## 2. Real 24-hour soak

Run the actual daemon against a staging Linear project. Prerequisites:

- `LINEAR_API_KEY` for a staging workspace — **never run against a live
  project with real issues**. The daemon WILL flip labels, post
  comments, and modify the bound issues.
- A staging team id in `LINEAR_TEAM_ID`.
- `shamu:ready`, `shamu:in-progress`, `shamu:review`, `shamu:blocked`
  labels present on the staging team.
- A publicly-reachable webhook URL. The simplest option is
  `shamu linear tunnel` (which wraps cloudflared).
- `LINEAR_WEBHOOK_SECRET` set + registered in Linear's webhook
  configuration with the same value.

Launch:

```
LINEAR_API_KEY=... \
LINEAR_TEAM_ID=... \
LINEAR_WEBHOOK_SECRET=... \
shamu linear serve \
  --webhook-port=7357 \
  --drain-timeout=600
```

Keep the daemon running for ≥24h. Periodically push a synthetic
`shamu:ready` label-add from the staging Linear UI and verify:

- The issue flips `ready → in-progress → review` (or `blocked` on
  synthetic failure).
- The rolling comment is updated with per-node checkpoints.
- Broker lifecycle log lines appear under `linear runtime` at DEBUG.
- `shamu.db` size grows linearly with flow_runs (no unexpected growth).

### Metrics to collect

Against a staging run, capture from the daemon's NDJSON log:

- `linear runtime: flow-shim prune` every ~50 pickups — `removed` count
  should stay bounded.
- `linear runtime: rate-limit overflow` — should never fire if the
  operator isn't flooding the webhook.
- Process RSS + heapUsed at 0h / 6h / 12h / 18h / 24h — should all stay
  within ±30% of the 0h baseline.

### Cleanup

After the 24h run:

1. `kill -SIGTERM $(pgrep -f 'shamu linear serve')` — the daemon enters
   drain; in-flight runs finish within `SHAMU_DRAIN_TIMEOUT`.
2. Verify the daemon exit code is `0`.
3. Close the cloudflared tunnel.
4. Rotate any credentials that appeared in logs if necessary.

### Known gaps (followups for 8.C)

- Broker env threading: the per-run `HTTPS_PROXY`/`HTTP_PROXY` env is
  generated by `withEgressBroker` but **not yet** consumed by the
  canonical flow's adapter runners. Live-vendor traffic still bypasses
  the broker until the flow's `makeSpawnOpts` threads `env` through.
  The broker's lifecycle (start / shutdown / event route) is already
  exercised per run.
- Policy sourcing: the daemon currently derives its policy from the
  union of `DEFAULT_ALLOWLISTS`. Operator-specific narrower policies
  require `--egress-policy <path>` plumb-through on `shamu linear
  serve`; tracked as follow-on.
- A 24h soak against the canonical flow's real adapters will want
  `shamu.config.ts` schema support for per-adapter allowlists — also
  follow-on.
